Admin Portal — 3 Approaches

Pick the architecture that fits best. Each affects how auth, routing, and roles are handled.

A

NextAuth.js + Credentials

Industry-standard Next.js auth library. Handles sessions, CSRF, cookies out of the box. Credentials provider for username/password now — Google/GitHub OAuth later with minimal changes. Admin routes protected by Next.js middleware.

Pros

  • OAuth-ready for future
  • Less code to maintain
  • Session management handled

Cons

  • Extra dependency
  • Some NextAuth config overhead

Recommended

B

Custom JWT + Middleware

Roll your own login endpoint that issues a signed JWT stored in an httpOnly cookie. Next.js middleware verifies the token on every admin request. No third-party auth library.

Pros

  • No extra dependencies
  • Full control over token shape

Cons

  • More code to write and maintain
  • OAuth requires significant extra work later
  • Easy to get CSRF/session wrong
C

Separate Admin App

Build the admin portal as a completely separate Next.js app or route group with its own auth, deployed independently. Main app stays unchanged.

Pros

  • Cleanest separation
  • Zero risk of admin code affecting public pages

Cons

  • Shares the same DB anyway
  • Two repos/deployments to manage
  • Overkill for this scale
